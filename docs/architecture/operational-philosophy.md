# Operational Philosophy

The platform should help a small county team operate AI governance with confidence, not overwhelm them with enterprise theater.

## Principles

- Governance first.
- Evidence always.
- Findings before dashboards.
- Explainability over magic.
- Operational clarity over novelty.
- Boring infrastructure over fragile complexity.
- Scanner orchestration over scanner reinvention.
- Human review where rights, safety, or legal exposure exists.

## What Operators Need

Operators need to answer:

- Which AI systems exist?
- Which systems are public-facing?
- Which systems affect rights, safety, benefits, law enforcement, employment, or access to services?
- Which systems use PII, PHI, CJIS data, or sensitive proxies?
- Which findings are critical or overdue?
- Which deployments are blocked?
- Which reviews are pending?
- What evidence supports each decision?
- What changed since the last review?

## Operational Tone

The application should feel calm, authoritative, audit-friendly, and direct. It should avoid playful AI aesthetics, chat-first framing, and vague confidence scores. The operator should feel like they are using a security and governance console for high-consequence public services.

## Small-Team Design

This is intended for one or two operators, so every feature should reduce cognitive load:

- Few core queues.
- Clear prioritization.
- Explicit ownership.
- Visible SLA state.
- Clear review state.
- Searchable evidence.
- Repeatable reports.
- Minimal infrastructure burden.

## Decision Bias

When uncertain, prefer:

- One maintainable module over three clever services.
- One clear workflow state machine over ad hoc labels.
- One normalized finding schema over tool-specific table sprawl.
- One evidence model over scattered attachments.
- One operational dashboard over many decorative charts.
